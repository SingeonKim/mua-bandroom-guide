# MUA 합주실 가이드 사이트 — 설계 문서

> 작성일: 2026-05-31
> 상태: 설계 확정 (구현 plan은 하네스 구성 후 별도 작성)
> 밴드: 무아(MUA) / 회사 공용 합주실 PA·모니터 세팅 문서화

---

## 1. 배경 & 목표

회사 공용 합주실의 PA/모니터 세팅을 전담 관리자 없이 예약제로 운영하다 보니 세팅이 꼬여,
작성자가 총대 메고 믹서(Behringer X32 Producer) 세팅값을 실측·확정하며 정리하는 중이다.

이미 완성한 **작성자용 `.md` 정리 문서**를 GitHub Pages로 발행하되, 두 독자층으로 분리한다.

- **spec (스펙 문서)**: 관리자/작성자용. 게인값·연결 케이스·매뉴얼 근거 등 상세. → 기존 `.md` 거의 그대로.
- **guide (사용 가이드)**: 일반 이용자용. "내 악기 어디 꽂나 / 게인 만지지 마라 / 씬 불러오는 법 / 문제 생기면" 수준으로 짧고 쉽게. → 신규 작성.

**우리 밴드(MUA)가 아니더라도** 합주실을 이용하는 사람이 빠르게 확인할 수 있는 가이드가 핵심 목적.

## 3. 레포 구조

기존 제안(`_config.yml` + `docs/`)은 Jekyll용이므로 MkDocs 구조로 대체한다.

```
mua-bandroom-guide/
├─ mkdocs.yml                    # 사이트 설정(테마/네비/검색) — Jekyll의 _config.yml 대응
├─ README.md                     # 레포 설명 + 로컬 미리보기/배포 방법
├─ docs/
│  ├─ index.md                   # 랜딩: "여기가 뭔지" + guide/spec 갈림길
│  ├─ guide.md                   # 일반 이용자용 (신규 작성, 짧고 쉽게)
│  └─ spec.md                    # 관리자/나용 (기존 .md 거의 그대로 이식)
└─ .github/workflows/deploy.yml  # main 푸시 시 mkdocs build → Pages 배포
```

> 참고: 설계/작업 문서(`docs/superpowers/`)는 사이트 빌드 대상에서 제외한다
> (mkdocs.yml의 `docs_dir`를 명시하거나 nav에 포함하지 않음).

## 4. 콘텐츠 설계

### 4-1. `spec.md` — 스펙 문서 (관리자/나용)

- 작성자가 이미 완성한 `.md` 전체를 **거의 그대로 이식**. 내용 완성도가 높아 재작성하지 않는다.
- **확정 / 잠정 / 빈칸** 3단계를 MkDocs Material에 맞게 시각적으로 구분:
  - 잠정값(혼자 확인, 합주 재검증 필요) → `!!! warning "잠정 — 합주 재검증 필요"` admonition
  - 빈칸 `______`(현장에서 채울 것) → `!!! note "현장에서 채울 것"` 또는 표기 유지
  - 확정값(실측) → 본문 표 그대로
- 긴 단일 페이지 + 우측 자동 목차(0~8장 + 작업 순서 체크리스트).
- 검색으로 "Rumble", "게인", "Bus13" 등 키워드 즉시 점프.

원본 문서 구성(그대로 유지):
0. 작업 전 필수 — 백업
1. 시스템 구성 요약 / 신호 흐름
2. 입력 채널 설정
2-1. 기타/베이스 앰프 — 연결 케이스 & 게인 기준(A/B/C/D)
2-2. 보컬 마이크 세팅 (PGA57 ×3)
3. 모니터 분배 (Bus1~5, Pre-fader)
4. 메인 스피커 (PRX ONE, Post-fader)
5. 보컬 리버브 (Bus13) — 확인 후 확정
6. 부밍/하울링 처리
7. TODO
8. 마무리 — 저장 & 기록
+ 작업 순서 요약 체크리스트

### 4-2. `guide.md` — 사용 가이드 (신규, 일반 이용자용)

핵심 원칙: **"씬 불러오기 + 본인 자리에 꽂기"만 알면 끝.**
잠정값·게인 수치는 일절 넣지 않는다 (이용자는 검증 안 된 값을 알 필요 없음).

구성:
1. **시작 전 1분** — "절대 게인/라우팅 건드리지 마세요. 끝나면 씬 다시 불러오기."
2. **씬 불러오는 법** — X32에서 기본 씬 로드 단계 (순서/스크린샷 자리)
3. **내 악기 어디 꽂나** — 표준 배치표(베이스→ch8, 키보드→9~12, 기타→13/14, 보컬→17~19) + 본인 모니터 위치
4. **외부 밴드라면 (최소 응용)** — "기타는 앰프 인풋에 꽂기 / 자리·게인 재배치가 필요하면 직접 만지지 말고 관리자에게" 수준의 1섹션
5. **소리가 안 나와요 / 문제 생기면** — 빠른 체크리스트(뮤트? 페이더? 씬 다시 로드) + 관리자 연락
6. **하지 말 것** — 게인·라우팅·48V 손대기 금지 (빨간 admonition 박스)

→ guide에 들어갈 정보는 spec의 **확정 정보만** 추려온다. 잠정/빈칸은 제외.

### 4-3. `index.md` — 랜딩

- 한 줄 소개("MUA 합주실 PA 안내")
- 큰 갈림길 2개: **［처음 오셨나요 → 사용 가이드］** / **［세팅 관리 → 스펙 문서］**
- 장비 한 줄 요약(X32 Producer / DHR12M ×5 / PRX ONE ×2)

## 5. 배포 설계 (`.github/workflows/deploy.yml`)

- 트리거: `main` 브랜치 푸시
- 동작: Python 환경 → `pip install mkdocs-material` → `mkdocs build` → Pages artifact 업로드 → 배포
- 사전 1회 설정: 레포 Settings → Pages → Source = "GitHub Actions"

## 6. 범위 밖 (YAGNI)

- 외부 밴드의 임의 장비/연결방식 완전 대응 (가이드는 표준 + 최소 응용까지만)
- 다국어 (한글 전용)
- spec 장별 페이지 분리 (한 페이지 + 목차로 충분)
- 잠정값을 guide에 노출 (의도적으로 제외)
