# docs-integrity-qa — 문서 무결성 검증

이 프로젝트의 1순위 리스크는 **spec의 미검증 잠정값/빈칸이 guide로 새어 이용자가 잘못 따라 하는 것**이다. 검증의 핵심은 단순 존재 확인이 아니라 **두 문서를 동시에 읽고 대조하는 경계면 교차검증**이다.

> 판단 기준은 **"확정이냐 미검증 잠정이냐"**다 — "게인 수치냐 아니냐"가 아니다. 확정된 운영값(케이스별 게인, 앰프 노브 위치)은 guide에 있어도 정상이다. **미검증 잠정값**(spec에서 `!!! warning "잠정"`로 표시된 것)과 **빈칸**(`______`)만 누출로 본다.

## 검증 순서 (점진적)

전체 완성 후 1회가 아니라 각 문서 완성 직후 돌린다. spec 완료 → guide 완료 → infra 완료 순.

### 1. 누출 교차검증 (최우선)

spec-editor의 **"전달 금지(미검증·잠정·빈칸)" 목록**을 기준선으로 삼는다. 확정 게인값은 누출이 아니다.

```bash
# guide/index에서 잠정·빈칸 표기 1차 스캔 (게인 수치 자체는 정상일 수 있음)
grep -nE '잠정|______' docs/guide.md docs/index.md
# spec에서 "잠정"으로 표시된 구체값이 guide에 그대로 등장하는지 교차 확인
```

- 핵심은 **spec에서 `!!! warning "잠정"`로 감싼 값**(모니터 분배 -10/-15, 메인 페이더, 메인 본체 -10dB, 리버브 B 구조)이 guide/index에 나타나는지다. 나타나면 **FAIL** — 파일·라인·해당 값·근거를 적어 guide-writer에 수정 요청.
- **확정 운영값은 통과**: 케이스별 확정 게인(앰프 인풋 0 / 멀티 직결 -10 / Return +10), 베이스 +21dB·마스터 12시, 앰프 노브 위치는 guide에 있어도 정상이다.
- `______` 빈칸이 guide에 그대로 노출되면 FAIL (단, 씬 이름·연락처처럼 의도적으로 비워 둔 현장 기입 자리는 허용 — 맥락으로 구분).

판단이 애매하면 **보수적으로 누출 위험 플래그** + 사람 확인 요청.

### 2. 빌드 검증

```bash
pip install mkdocs-material >/dev/null 2>&1
mkdocs build --strict
```

- `--strict`는 깨진 내부 링크·없는 nav 페이지를 에러로 잡는다.
- 설치 불가 시: YAML 문법 + nav 참조 경로 존재만 정적 점검하고 "로컬 빌드 미수행"을 리포트에 **명시**(은폐 금지).

### 3. 링크/네비 정합성

- index.md의 guide/spec 링크가 실제 파일을 가리키는지.
- mkdocs.yml `nav`의 모든 항목 파일이 존재하는지.
- `docs/superpowers/` 설계 문서가 사이트 nav/검색에 노출되지 않는지.

### 4. 콘텐츠 일관성

- spec의 확정 ch 매핑과 guide의 표준 배치표가 모순 없는지 대조.

## 리포트 형식

```
[항목] PASS / FAIL
  - FAIL 시: 파일:라인 / 발견 내용 / 근거 / 수정 제안
```

- 빌드 도구 미실행은 반드시 명시한다.
- 발견을 숨기지 않는다 — FAIL도 PASS도 명확히.

## 협업

- spec-editor: 잠정값 목록 수신
- guide-writer: 누출 FAIL 통보(파일·라인·수치)
- site-infra: 빌드 FAIL 로그 전달
